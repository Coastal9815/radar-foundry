# MRW Lightning — Architecture Clarification

**Status:** Pre-implementation clarification. No code, no UI changes.

**Purpose:** Tighten four aspects of the lightning architecture before build: time standard, strike identity/provenance, machine roles, and canonical vs derived separation.

---

## 1. Canonical Time Standard

**Decision:** The normalized strike model uses **UTC** as the primary time standard internally. Local time (America/New_York) is a display concern only.

**Rationale:**
- UTC avoids timezone edge cases when merging realtime and archive paths, deduplicating, or replaying across day boundaries.
- Replay and archive products are timezone-agnostic; consumers apply local formatting.
- Consistent with common practice for timestamped sensor data.

**Implications:**
- Canonical strike record: `timestamp` in ISO 8601 UTC (e.g. `2026-03-14T18:23:45.123Z`).
- All internal processing, storage, and product generation use UTC.
- Downstream displays (Master Dashboard, website, players) format to America/New_York for human-readable output. That formatting happens at the presentation layer, not in the canonical model or products.

## 2. Strike Identity and Provenance

The canonical strike record must support deduplication, source tracing, replay/debugging, and safe merging of realtime and archive paths. The following provenance and identity fields are required.

### 2.1 Strike Identity (Deduplication)

- **Strike ID** — A deterministic identifier derived from the physical strike. Same strike from two sources (e.g. realtime tail and archive extraction of the same day) must produce the same ID so we can deduplicate when merging. Recommended: content hash or composite key of `(timestamp_utc, distance_mi, bearing_deg, sensor_id)`. Source-provided IDs (if any) are unreliable across sources; a content-derived ID is source-agnostic and stable.

### 2.2 Provenance (Source Tracing)

- **Source** — Which adapter produced this record: `nex_realtime`, `nex_archive`, `flashgate`, `nxutil`. Required for debugging and understanding data lineage.
- **Source reference** — Optional but valuable. For .nex: the file and byte offset (e.g. `20260314.nex@284512`). For archive extraction: the file path or date. For nxutil: the source CSV/file. Enables tracing a strike back to its origin for reprocessing or format validation.
- **Ingested at** — UTC timestamp when MRW ingested this record. Useful for pipeline latency analysis and debugging.

### 2.3 Raw Payload (Reprocessing)

- **Raw** — Original source values (raw distance, bearing, timestamp bytes, etc.). Preserved for reprocessing when we improve parsers or fix bugs. Optional in published products; essential in internal pipeline and archive storage.

### 2.4 Merging Realtime and Archive

When both realtime (tail-.nex) and archive (full-file) paths process the same day, they will produce overlapping strikes. The strike ID allows deduplication: before appending to the strike stream or product, check if a strike with that ID already exists. Keep one; discard the duplicate. Source and source_ref help diagnose when duplicates occur and which path "won."

## 3. Machine Role Refinement

### Model A

| Machine | Role |
|---------|------|
| Lightning-PC | Acquisition |
| wx-core | Primary processing (ingestion, normalization, product generation, animation) |
| wx-i9 | Secondary processing / rendering support |
| pi-wx | **Final shared product host / API publisher** |

### Model B

| Machine | Role |
|---------|------|
| Lightning-PC | Acquisition |
| wx-core | Ingestion + processing |
| wx-i9 | Publisher |
| pi-wx | Optional / unused |

### Recommendation: Model A

**Rationale:**

MRW already uses pi-wx as the canonical host for shared JSON products. Station data (now.json, wind.json, rain.json, air.json, climatology.json, etc.) is generated by pi-wx generators and published from `/home/scott/dashboard/data/`. The public API (api.moonriverweather.com/data/) serves these products. The Master Dashboard loads station data via the /pi-wx-data proxy (wx-i9 → pi-wx). The website consumes from the same API.

Lightning products are shared JSON outputs, like air.json and now.json. They should follow the same pattern: **processed elsewhere, published from pi-wx**. That gives:

- **Single source of truth** — All MRW JSON products (weather + lightning) live on pi-wx. Website and dashboards consume from one place.
- **Consistent API** — api.moonriverweather.com/data/lightning_rt.json, lightning_recent.json, etc. Same pattern as air.json.
- **Unified consumption** — Master Dashboard already proxies /pi-wx-data for station data; lightning products would flow through the same path.

wx-i9 remains the publisher for **binary/raster products** (radar frames, satellite frames, overlay frames). Those are large, frame-based, and served directly from wx-i9. JSON products belong on pi-wx.

**Refined flow:** wx-core processes lightning → syncs JSON products to pi-wx → pi-wx hosts and publishes via API. wx-core may also produce overlay frames and sync those to wx-i9 for serving (same as radar). wx-i9 serves frames; pi-wx serves JSON.

## 4. Canonical Strike vs Derived Products

### 4.1 Fields in the Canonical Strike Record Only

These belong **only** in the canonical strike. They are the atomic truth.

| Field | Purpose |
|-------|---------|
| timestamp (UTC) | When the strike occurred |
| distance_mi, bearing_deg | Sensor-relative location |
| lat, lon | Geographic location (computed) |
| strike_id | Deterministic identity for deduplication |
| source | Adapter that produced this record |
| source_ref | Optional; file/offset for tracing |
| ingested_at | Optional; when MRW ingested (UTC) |
| raw | Optional; original payload for reprocessing |
| strike_type, polarity | Optional; if source provides (CG/IC, +/-) |

### 4.2 Fields That Exist Only in Derived Products

These are **never** stored on the canonical strike. They are computed and stored in derived products.

| Field / Concept | Derived Product | Rationale |
|-----------------|-----------------|------------|
| Cell membership, cell_id | lightning_cells.json | Cells are clusters; membership is derived by clustering. Strike does not "know" its cell. |
| Alert state, triggered_alert | lightning_alerts.json | Alerts are evaluated from strikes; the alert object holds trigger info. Strike does not store alert state. |
| Age, opacity, color, size | Overlay frames, client display | Computed from timestamp at render time. Not stored. |
| Frame index, frame path | Replay manifest | Replay organizes strikes into frames; the manifest describes frames, not strikes. |
| Strike rate, count | lightning_alerts.json, lightning_stats | Aggregated; not per-strike. |
| Local-time display string | Client | Formatting of timestamp for display; not stored in products. |

### 4.3 Principle

The canonical strike is **minimal and immutable**: it describes the physical strike and its provenance. Derived products add interpretation (cells, alerts), aggregation (rates, counts), or presentation (frames, manifests). Consumers that need cells use lightning_cells; consumers that need alerts use lightning_alerts. The strike record itself stays clean.
